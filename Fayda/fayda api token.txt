Introduction
VeriFayda 2.0 (esignet) enables relying parties to authenticate users and securely access cross-audience resources using:

OpenID Connect (OIDC) for user authentication (Authorization Code + PKCE)

Token Exchange for token interoperability across source audience → target audience

Introspection at the resource server to validate the resulting (often opaque) token

Prerequisites

Knowledge of OAuth 2.0 and OpenID Connect.

Access to VeriFayda 2.0(eSignet) credentials and onboarding details:

     - Client ID

     - Redirect URI

     - VeriFayda 2.0 (eSignet) endpoints (authorization, token, userinfo, token exchange, consent, introspection)

     - Private key for private_key_jwt (Base64-encoded JSON JWK)

A web application stack (browser app + backend, or a backend-only implementation).

1. Overview of the Interoperability Flow

Core concepts

Authorization Code Flow with PKCE

code_verifier: high-entropy secret generated by the client

code_challenge: BASE64URL(SHA256(code_verifier))

PKCE prevents authorization code interception/replay

Client assertion (private_key_jwt)

A JWT signed with the client’s private key (no client_secret)

Used to authenticate the client at the token endpoint and token exchange endpoint

Source audience vs target audience

Source audience: the client/application context that initially obtained the access token via OIDC

Target audience: the audience for the downstream resource server you want to call

Consent

Token exchange may require explicit user consent the first time (or for new scopes/audiences)

Opaque access token + introspection

Token exchange often returns an opaque token; the resource server validates it via introspection

 2. Steps for Integration

 2.1 Authorization Request (OIDC + PKCE)

2.1.1 Generate PKCE values

Generate a random code_verifier (per login transaction).

Derive code_challenge using S256:

code_challenge = BASE64URL(SHA256(code_verifier))

Store the code_verifier securely (server-side session or secure client storage) until the callback step.

Snippet: Example PKCE generation (JavaScript)



const codeVerifier = randomUrlSafeString(64);
const codeChallenge = base64UrlEncode(sha256(codeVerifier));
2.1.2 Redirect to authorization endpoint

Send the user to the authorization endpoint with these required parameters:



- client_id
- response_type=code
- redirect_uri
- scope (examples: openid, or openid profile email)
- state (CSRF protection)
- code_challenge
- code_challenge_method=S256
Optional parameters commonly used:

acr_values: request a specific authentication context

       Example: OTP only: mosip:idp:acr:generated-code

       Example: OTP or biometrics: mosip:idp:acr:generated-code:biometrics

claims_locales: languages for localized KYC payload fields (example: en am)

claims: request specific claims and mark some as essential



 {AUTHORIZATION_ENDPOINT}?
  client_id={CLIENT_ID}&
  response_type=code&
  redirect_uri={REDIRECT_URI}&
  scope={SCOPE}&
  state={STATE}&
  code_challenge={CODE_CHALLENGE}&
  code_challenge_method=S256&
  acr_values={ACR_VALUES}&
  claims_locales={CLAIMS_LOCALES}&
  claims={URL_ENCODED_JSON_CLAIMS}

2.2 Callback Handling

After successful authentication, VeriFayda2.0 (eSignet) redirects the user to your redirect_uri with:

code 

state

What must be done 

- Validate state matches what you issued. 

- Exchange code at the token endpoint using: 

          - code_verifier (PKCE) 

          -  client_assertion (private_key_jwt)

2.3 Token Request (Authorization Code → Tokens)

Send POST to the token endpoint using:

Header: Content-Type: application/x-www-form-urlencoded   

 Body parameters:                                       

     - grant_type=authorization_code

     - code

     - redirect_uri

     - client_id

     - client_assertion_type=urn:ietf:params:oauth:client-assertion-type:jwt-bearer

     - client_assertion={SIGNED_JWT} 

     - code_verifier

Snippet: Token request body



grant_type=authorization_code&
code={AUTHORIZATION_CODE}&
redirect_uri={REDIRECT_URI}&
client_id={CLIENT_ID}&
client_assertion_type=urn:ietf:params:oauth:client-assertion-type:jwt-bearer&
client_assertion={SIGNED_JWT}&
code_verifier={CODE_VERIFIER}
If successful, you receive:

     - access_token (this is your source-audience access token)

     - id_token (optional depending on scopes/config)

     - token_type, expires_in

2.3.1 Client Assertion (private_key_jwt)

Private key format

  The private key is typically provided as:

- Base64-encoded UTF-8 JSON containing an RSA JWK

Client assertion JWT claims

- iss: your client_id

- sub: your client_id

- aud: token endpoint URL

- iat: issued-at time

- exp: expiration time (short-lived; e.g., 5–15 minutes is typical)

Snippet: Example implementation (Node.js, jose)



import { SignJWT, importJWK } from "jose";
export async function generateClientAssertion({ clientId, tokenEndpoint, privateKeyBase64 }) {
  const jwkJson = Buffer.from(privateKeyBase64, "base64").toString("utf8");
  const jwk = JSON.parse(jwkJson);
  const privateKey = await importJWK(jwk, "RS256");
  return await new SignJWT({})
    .setProtectedHeader({ alg: "RS256", typ: "JWT" })
    .setIssuer(clientId)
    .setSubject(clientId)
    .setAudience(tokenEndpoint)
    .setIssuedAt()
    .setExpirationTime("2m")
    .sign(privateKey);
}
Snippet: Example implementation (Python)



import base64
import json
import jwt
from datetime import datetime, timedelta, timezone
from jwcrypto import jwk
def generate_client_assertion(client_id: str, token_endpoint: str, private_key_base64: str) -> str:
    jwk_json = base64.b64decode(private_key_base64).decode("utf-8")
    key = jwk.JWK.from_json(jwk_json)
    pem = key.export_to_pem(private_key=True, password=None)
    now = datetime.now(timezone.utc)
    payload = {
        "iss": client_id,
        "sub": client_id,
        "aud": token_endpoint,
        "iat": int(now.timestamp()),
        "exp": int((now + timedelta(minutes=2)).timestamp()),
    }
    return jwt.encode(payload, pem, algorithm="RS256", headers={"typ": "JWT"})
2.4 Retrieve User Info (UserInfo Endpoint)

Use the source-audience access token to call the userinfo endpoint:

Header: Authorization: Bearer {ACCESS_TOKEN}

Note on response format
Depending on deployment, the userinfo response can be:

 - JSON or

 - a JWT string (signed and/or encrypted), which you decode as a JWT

2.5 Token Exchange (Token Interoperability)
Token exchange is used to obtain a new token intended for a target audience (resource server audience).

Send POST to the token exchange endpoint using application/x-www-form-urlencoded with:

- grant_type=urn:ietf:params:oauth:grant-type:token-exchange

- subject_token_type=urn:ietf:params:oauth:token-type:access_token

- subject_token={SOURCE_AUDIENCE_ACCESS_TOKEN}

- audience={TARGET_AUDIENCE_CLIENT_ID}

- scope={TARGET_SCOPE}

- client_id={CLIENT_ID}

- client_assertion_type=urn:ietf:params:oauth:client-assertion-type:jwt-bearer

- client_assertion={SIGNED_JWT}

Snippet: Token exchange request body 



grant_type=urn:ietf:params:oauth:grant-type:token-exchange&
subject_token_type=urn:ietf:params:oauth:token-type:access_token&
subject_token={SOURCE_AUDIENCE_ACCESS_TOKEN}&
audience={TARGET_AUDIENCE_CLIENT_ID}&
scope={TARGET_SCOPE}&
client_id={CLIENT_ID}&
client_assertion_type=urn:ietf:params:oauth:client-assertion-type:jwt-bearer&
client_assertion={SIGNED_JWT}
Responses
Success
- access_token: often opaque (target-audience token)

-  token_type, expires_in, scope

Consent required
  You may receive:

   - error=consent_required

   - consent_uri

In that case:

Redirect the user to consent_uri

After the user approves, record consent if your deployment requires it

Retry token exchange

2.6 Resource Server Validation (Introspection)

When calling a resource server using the exchanged token:

The client sends: Authorization: Bearer {TARGET_AUDIENCE_ACCESS_TOKEN}

The resource server validates it by calling the introspection endpoint:

          - token={TARGET_AUDIENCE_ACCESS_TOKEN}

          - token_type_hint=access_token

          - client_id={TARGET_AUDIENCE_CLIENT_ID_USED_FOR_INTROSPECTION}

Snippet: Introspection request 



token={TARGET_AUDIENCE_ACCESS_TOKEN}&
token_type_hint=access_token&
client_id={TARGET_AUDIENCE_CLIENT_ID_USED_FOR_INTROSPECTION}
If introspection returns "active": true, the resource server may return the protected response.

3. Error Handling

3.1  Authorization endpoint errors

Common errors (examples):

-invalid_request: missing/invalid parameters

-invalid_transaction: transaction interrupted or expired

-invalid_assertion: client assertion JWT issues

-consent_required: user must complete consent flow

3.3 Troubleshooting invalid_assertion

Check:
         - JWT signature is valid for the registered public key

         - Claims match expectations:

              -iss and sub are your client_id

              -aud is the correct endpoint URL

              - iat and exp are valid (not expired; not too far in the future)

         - The correct key and algorithm (RS256) are used

4. Rendering User Info (optional)

After retrieving (and optionally decoding) userinfo, you can render it in your application UI. The exact rendering is application-specific; the critical integration requirement is that userinfo retrieval is authorized by the source-audience access token obtained from the OIDC flow.

5. Reference terms

 OAuth 2.0 Token Exchange

        - An IETF(Internet Engineering Task Force) standard that defines how to exchange one token for another.  

        - In this interoperability flow, it is used to exchange a source-audience access token (subject_token) for a new access token that is valid for a target audience (audience).

subject_token

          - The token you currently have and want to exchange (the input token).

audience (target audience)

    - The audience value that indicates which downstream resource server the exchanged token should be valid for.

Opaque access token

   - Opaque tokens are typically validated using introspection.   

Introspection

A server-to-server call from the resource server to VeriFayda 2.0 (eSignet) to validate an access token.
